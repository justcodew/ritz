# I built the same software 3 times, then Rust showed me a better way

> 作者：Jeffrey Rennie
> 平台：ITNEXT
> 原文：<https://itnext.io/i-built-the-same-software-3-times-then-rust-showed-me-a-better-way>

我开发的是处理 PDF 文档的软件。在二十多年的时间里，我用很多不同的方式构建过这类软件：

- 用 Python + PLY
- 用 C++ + Flex 和 Bison
- 用 C++ 手写递归下降解析器
- 用 Rust 手写递归下降解析器

Rust 实现比 C++ 实现快了 3 倍，因为 Rust 向我展示了在内存中表示 PDF 文档的全新方式。

在用 Rust 构建第四个应用时，我尝试用 C++ 中相同的方式来表示 PDF 文档的内存结构，但 Rust 不允许我这么做。无论我怎么尝试，Rust 总是报出生命周期和可变性方面的错误。我花了大约一周时间，试图说服 Rust 编译器让我的代码编译通过，但编译器寸步不让。最终我放弃了，转而重新设计了 PDF 文档的内存表示，使其符合 Rust 的规则。结果是性能大幅提升，内存占用也大幅降低。

需要澄清的是，我并不认为 C++ 和 Rust 应用之间的性能差异是语言本身造成的。如果我在 C++ 中也采用和 Rust 一样的方式来表示 PDF 内部结构，我敢打赌 C++ 代码会比 Rust 更快。我只是从没想过要那样写 C++ 代码。

## PDF 文档内部结构

PDF 文档的内部结构很复杂。ISO 32000-1 规范（对应 PDF 1.7）足足有 756 页，这甚至还没包括完整解码 PDF 所需的大量辅助技术，比如完整的 JPEG 解码器、多种字体文件解码器、压缩算法等。

然而，PDF 文档的核心骨架其实相当简单，和 JSON 差别不大。考虑下面这个表示家谱的 JSON：

```json
{
  "id": 0,
  "name": "Michael Doe",
  "birthday": [1995, 3, 10],
  "father": {
    "id": 1,
    "name": "John Doe"
  },
  "mother": {
    "id": 2,
    "name": "Jane Smith"
  }
}
```

我将把这个 JSON 转换成更像 PDF 文档的形式，并解释转换过程：

- `id` 字段已被移除，替换为唯一的**对象编号**。`id` 由隐式 JSON schema 定义，但对象编号由 PDF 规范定义。
- 指向其他对象的链接由一个符号表示：`R-对象编号`。
- 文件末尾的**查找表**列出了每个对象的起始行号。这使得从一个对象链接到另一个对象变得快速而简单。

你会如何在内存中表示这种文件格式？要知道大多数 PDF 文档都太大而无法完全放入内存，而且文档及其对象只会被单线程访问。

## C++ 实现

在 C++ 中，我是这样表示 PDF 文档的：

```cpp
class Object {
public:
  virtual void write(std::ostream&) const = 0;
  virtual ~Object() {}
private:
  int refcount;
};

class String : public Object { ... };
class Number : public Object { ... };
class Array : public Object { ... };
class Dictionary : public Object { ... };
class Link : public Object { ... };
```

这段代码遵循面向对象设计原则。每种对象类型（字符串、数组、链接等）都有自己的类，继承自 `Object` 基类。每个类都知道如何将自身写入 PDF 文件。所有对象都在堆上分配。引用计数确保对象只要还被需要就保留在内存中。

让我们计算销毁下面这个对象所需的操作：

```json
{
  "name": "Michael Doe",
  "birthday": [1995, 3, 10],
  "father": "R-0",
  "mother": "R-1"
}
```

- 释放 8 个对象（3 个数字 + 2 个链接 + 1 个数组 + 1 个字符串 + 1 个字典）
- 减少文档的引用计数 8 次
- 减少子对象的引用计数 7 次
- 释放 5 个字符串
- 释放 1 个 vector
- 释放 1 个哈希表

总共需要 **15 次释放** 和 **13 次引用计数递减**。对象可能散布在堆中的各处，缓存局部性较差，这会负面影响性能。在这个上下文中，递减操作之后还有一个代码分支。代码分支同样对性能不利。

这段 C++ 代码无法直接翻译成 Rust，因为 Rust 不允许这样随意使用指针。看看这段 C++ 代码，你能回答下面这个问题吗：文档的存活时间更长，还是文档缓存中某个对象的存活时间更长？仅凭我提供的代码是无法判断的，而 Rust 要求对这个问题的答案有明确的界定，否则它将拒绝编译。

我可以尝试使用 `Rc<>`（一种引用计数智能指针）来满足 Rust 编译器的要求。但我在写这段代码时 Rust 还是新手，并不知道或没有去寻找引用计数智能指针，因为我觉得引用计数会牺牲掉 Rust 借用检查器一半的好处——将编译期保证降级为运行时检查，那就跟 C++ 中的 `assert` 语句没什么区别了。那写 Rust 还有什么意义呢？直接用 C++ 写反而更容易。

当然，用 Rust 的 trait 来实现类似 `Object` 基类也是可能的，但我选择使用了 `enum`。Rust 的 `enum` 类似于 C++ 的 `std::variant<>`，但 Rust 提供了更简单的语法，让使用 `enum` 变得容易且不易出错。无论如何，Rust 的 `enum` 对我来说是个新鲜事物，我很想尝试一下。

## Rust 实现

```rust
enum PdfValue {
    Null,
    Bool(bool),
    Integer(i64),
    Real(f64),
    String(String),
    Name(String),
    Array(Vec<PdfValue>),
    Dict(HashMap<String, PdfValue>),
    Stream {
        dict: Box<PdfValue>,
        data: Vec<u8>,
    },
}
```

Rust 的 `lookup()` 函数会返回对象的一份拷贝，而 C++ 版本返回的是指向缓存中对象的指针。我本以为拷贝对象会显著拖慢性能，但基准测试和性能分析工具告诉我，使用 `enum` 带来的性能收益远远超过了拷贝对象的成本。由于对象不再原地修改，我添加了一个 `set()` 函数来用对象的修改更新文档。

与 C++ 代码相比，Rust 代码在堆上分配的内存块要少得多。

让我们重新计算销毁前面那个对象所需的操作：

- 释放 1 个 `Vec<>`
- 释放 1 个 `HashMap<>`
- 释放 5 个字符串

总共只需 **7 次释放** 和 **0 次引用计数递减**。这与 C++ 的 15 次释放和 13 次引用计数递减形成了鲜明对比。

仅这一项改变，就已经让我的 Rust 程序性能超过了 C++ 版本的两倍。

减少堆内存分配带来的性能提升给了我很大启发，我把 Rust 代码中的字符串实现也换成了**短字符串优化（SSO，Small String Optimization）**类型：较短的字符串直接存储在栈上，不分配任何内存；只有较长的字符串才需要堆分配。这样，上面例子中的字符串键 `"name"` 和 `"birthday"` 就完全不产生堆分配。只有像 `"Michael Doe"` 这样较长的字符串才需要一次堆分配。

因此，销毁该对象的最终开销为：

- 释放 1 个 `Vec<>`
- 释放 1 个 `HashMap<>`
- 释放 1 个字符串

总共 **3 次释放** 和 **0 次引用计数递减**。相比 C++ 的 15 次释放和 13 次引用计数递减，堆分配和引用计数的急剧减少，加上缓存局部性的提升，让我的 Rust 实现最终比 C++ 实现快了约 **3 倍**。

## Rust 并不比 C++ 更快

要让 C++ 达到类似我 Rust 代码的性能表现是完全可能的，而且并不太难。事实上，我敢打赌，如果应用了同样的一系列优化，C++ 代码会比 Rust 更快。

但这并不是本文的重点。

本文的重点是：**在 Rust 中，那些显而易见、最便捷的写法，最终恰好就是性能最优的写法**。这太令人惊叹了。

我写 C++ 代码是在 2005 年左右，那时 `std::variant<>` 还不存在，`std::string` 也没有做短字符串优化，OOP 还是公认的最佳实践。我已经有大约十年没有密集地编写 C++ 了。如今 C++ 程序员的公认最佳实践是否已经改变？

在你最喜欢的编程语言中，有没有类似 Rust 中 `enum` 这样的特性？你觉得它能胜任这项任务吗？
